# Migrating from Horizon v0.3.x to v0.4

This guide documents the user-visible changes in Horizon v0.4 and what
existing v0.3 callers should know. Each section is owned by the track
that landed the change so other tracks can append cleanly. Every change
in this guide is additive — v0.3 programs and projects continue to build
unchanged.

## Remote imports: `m31labs.dev` meta-redirect (v0.4 — additive)

v0.3 wired remote imports for the `github.com/<org>/<repo>@<version>`
shape directly and documented `m31labs.dev/...` as a stub: the shape
was defined in ADR-0009 but the production HTTP discovery had not yet
landed, so a `m31labs.dev/...@<version>` import fell through to the
vendor walk and ultimately HZN1554. v0.4 implements the discovery, so
`m31labs.dev/<org>/<repo>@<version>` is now a working import path.

### What changed

A `m31labs.dev/<org>/<repo>` import with a `@<version>` suffix now
resolves its git clone URL via HTTP meta-redirect discovery rather than
failing. Resolution mirrors Go's import-path discovery:

1. Horizon issues `GET https://m31labs.dev/<org>/<repo>?horizon-import=1`.
2. It scans the response body for a tag of the form
   `<meta name="horizon-import" content="<prefix> git <clone-url>">`
   (three whitespace-separated fields: the import prefix, the literal
   `git`, and the clone URL).
3. The advertised `<clone-url>` is then handed to the same lockfile-pin
   + content-addressed cache path used by `github.com/...` imports.

```hzn
import audit "m31labs.dev/m31labs/horizon-audit@v0.4.0"
```

After discovery, `hzn get <path>@<version>` writes the `hzn.lock` entry
(resolved 40-char SHA + content sha256) exactly as it does for a
`github.com/...` import, and `hzn check` verifies the cached content on
every build. The lockfile entry's `path` is the original
`m31labs.dev/...` import path; the discovered clone URL is an internal
resolution detail and is not recorded in `hzn.lock`.

### New diagnostic

| Code | Trigger |
| ---- | ------- |
| HZN1705 | meta-redirect discovery failed — the host could not be reached, returned a non-200 status, or served no parseable `<meta name="horizon-import">` tag. Distinct from HZN1703 (git clone/fetch failure) and HZN1704 (bad version syntax). |

### No action required

This change is purely additive:

- `github.com/<org>/<repo>@<version>` imports resolve exactly as in
  v0.3 — no network round-trip is added for the github path.
- `m31labs.dev/...` imports that previously failed (HZN1554) now
  resolve; nothing that worked in v0.3 changes behavior.
- URL-shaped imports **without** `@version` still fall through to the
  `vendor/` walk.

### Documented limits (unchanged from v0.3)

- Only `m31labs.dev` is wired for meta-redirect discovery in v0.4.
  Other arbitrary HTTPS hosts still fall through to the vendor walk →
  HZN1554.
- `git+ssh://` paths and arbitrary-host discovery remain out of scope.
- `@latest` / `@HEAD` floating refs stay rejected (HZN1704) —
  reproducibility is non-negotiable.
- Real-network discovery in the test suite is gated behind
  `HORIZON_NETWORK_TESTS=1`; the default test run stays offline.

## Batch lockfile update (v0.4 — additive)

v0.3 shipped per-dependency lockfile writes through `hzn get
<repo>@<version>`. v0.4 adds a batch path: the new `-lockfile-update`
flag on `hzn check` resolves every `@version` import in the build and
rewrites `hzn.lock` in a single pass, then proceeds to verify against
the freshly-pinned content.

```sh
# After bumping one or more @version pins in your sources:
hzn check -lockfile-update ./probes
```

### What changed

- `hzn check -lockfile-update` runs the resolver in write-back mode,
  merges each newly-resolved entry into the existing `hzn.lock` (an
  unchanged pin is left as-is), writes the lockfile atomically, and then
  runs the normal verify check. Per-entry update lines print to stderr.
- Default `hzn check` (flag absent) is **unchanged** — it stays
  verify-only and never mutates `hzn.lock`.

### No action required

This change is purely additive. It complements `hzn get` for
version-bump workflows that touch several dependencies at once;
resolution diagnostics (HZN1703 fetch failure, HZN1704 bad version)
match `hzn get` exactly. A package with no remote imports is a clean
no-op — no `hzn.lock` is created.
