# Migrating to v0.2 Package Composition

## TL;DR

Horizon v0.2 turns the language from a "one `.hzn` file per program" tool into
a "vendor-style multi-package authoring surface". You can now split one
program across several `.hzn` files in the same directory and import
sibling or vendored packages by URL-shaped path. The build still produces a
single `capability/v1` manifest covering every reachable declaration.

Three things to know:

1. **Same-package multi-file builds are now first-class.** Drop multiple
   `.hzn` files into one directory; declare `package myname` at the top of
   each; references resolve across files. Duplicate declarations across
   files in the same package produce a clearer diagnostic that names the
   prior file.
2. **Cross-package imports resolve at compile time.** `import events "<path>"`
   resolves to a local-filesystem directory: a relative path (`./events`)
   or a vendored URL-shaped path (`./vendor/<full-path>/`). No remote
   fetching, no lockfile, no version resolution.
3. **Manifests gain an additive `origin` field on imported declarations.**
   Old readers ignore the field; the `schema` constant stays at
   `m31labs.dev/horizon/capability/v1`. Re-reading existing single-package
   manifests is unaffected.

This migration is **opt-in**: existing single-package builds compile
unchanged and produce byte-identical manifests.

## Import Path Shapes

The `import <alias> "<path>"` syntax is unchanged from v0.1; v0.2 grows
the resolver to understand four shapes:

| Shape                                       | Example                                          | Resolution                                                                 |
|---------------------------------------------|--------------------------------------------------|----------------------------------------------------------------------------|
| Built-in compiler namespace                 | `import bpf "m31labs.dev/horizon/runtime/kernel"` | Recognised; no file walking. The alias routes calls through compiler intrinsics. |
| Relative path                               | `import util "./internal/util"`                  | Resolved relative to the importing file's directory.                       |
| Absolute path                               | `import x "/abs/path/to/pkg"`                    | Resolved as a filesystem path. Emits `HZN1556` warning ("not portable").   |
| Vendored URL-shaped path                    | `import events "m31labs.dev/myorg/events"`       | Walks parent directories looking for `./vendor/<full-path>/`. Mirrors Go's pre-modules vendor pattern. |

Resolution stops at the build root (the directory passed to
`hzn check <dir>`). There is no `hzn.toml` workspace file in v0.2.

Built-in namespace aliases are interchangeable: `import bee "m31labs.dev/horizon/runtime/kernel"`
makes `bee.current_pid()` a legal call site. Any alias on a built-in path
resolves to the same compiler intrinsics.

A package layout under a resolved path is the same as today's
`hzn check <dir>`: every `.hzn` file in the directory (non-recursive; a
`vendor/` subdirectory is NOT walked into) is treated as a member of the
same package. The alias on the `import` line is the local name; the
imported package's declared `package <name>` is informational. Mismatches
produce `HZN1525` advisory.

### New diagnostics

| Code      | Severity | When                                                                                                  |
|-----------|----------|-------------------------------------------------------------------------------------------------------|
| `HZN1004` | error    | An import alias shadows a hardcoded compiler namespace (`import bpf "./mything"`). Extension of the existing top-level-collision code. |
| `HZN1554` | error    | Import path could not be resolved (no matching vendor directory, no matching relative path).          |
| `HZN1555` | error    | Import cycle detected at a package directory.                                                         |
| `HZN1556` | warning  | Absolute import path is not portable across machines.                                                 |
| `HZN1557` | error    | Qualified selector type references an unknown import alias.                                           |
| `HZN1558` | error    | Qualified selector type names a struct the imported package does not declare.                         |

## Capability Aggregation Rules

When a build spans multiple packages (root + imports), the aggregator
merges per-package contributions into one root manifest using the
following rules:

1. **Root-package capabilities keep bare names.** A `capability ExecObserve …`
   declared in the root package emits as `"name": "ExecObserve"` (after
   namespace resolution).
2. **Imported-package capabilities are qualified by their import alias.**
   A `capability ExecObserve` declared in a package imported as
   `events` emits as `"name": "events.ExecObserve"` and carries
   `"origin": "events"` in the manifest entry.
3. **Reachability gates emission.** A capability is emitted only if its
   declaring function is referenced from the root package's entrypoint
   set. Unreached capabilities from imported packages do NOT appear in
   the manifest.
4. **Identical contributions deduplicate silently.** Two manifests
   contributing the same qualified name with the same shape (danger,
   section, program, map access) merge into one entry.
5. **Same name, different shape → error.** Two packages contributing the
   same qualified capability name but disagreeing on danger / program /
   section / map access fire `HZN1560`.
6. **Same section value, different packages → warning.** Two packages
   independently claiming the same kernel-hook section string
   (e.g., `"kernel.process.exec.observe"`) fire `HZN1553`. Both entries
   surface in the aggregated manifest under their qualified names.
7. **Maps and types deduplicate by qualified name.** Conflicting shapes
   fire `HZN1566` (map shape) or `HZN1567` (type schema) at the
   manifest-aggregation layer. The IR-merge layer separately fires
   `HZN1564` (struct layout) and `HZN1565` (capability) when the
   collision is visible before per-package manifests are aggregated.
   The aggregator codes carry distinct `Suggest` strings that name the
   manifest-aggregation layer so a diagnostic identifies which layer
   detected the collision without forcing the reader to consult code.

Output is sorted lexicographically by qualified name within each section
(Capabilities, Maps, Types, Programs) so JSON marshalling is byte-stable
across invocations and across machines.

## Manifest Changes (Additive)

The manifest schema constant is unchanged:

```go
capability.SchemaV1 // "m31labs.dev/horizon/capability/v1"
```

Three additive `omitempty` fields appear on imported-package
contributions:

- `Capability.origin` — the importing alias (`"events"`).
- `Map.origin` — the importing alias.
- `TypeSchema.origin` — the importing alias.

A v0.1 reader that ignores unknown JSON fields parses v0.2 manifests
without modification. The shape of every other field is unchanged.

**Example: imported-execcount.** The example at
`examples/imported-execcount/` imports `events.ExecEvent` from a
vendored package and emits a manifest whose `types` entry carries
`"origin": "events"`:

```json
{
  "name": "ExecEvent",
  "origin": "events",
  "kind": "struct",
  "size": 32,
  "align": 8,
  "fields": [
    { "name": "ts_ns", "offset": 0, "type": "u64" },
    { "name": "pid",   "offset": 8, "type": "u32" },
    { "name": "uid",   "offset": 12, "type": "u32" },
    { "name": "comm",  "offset": 16, "type": "[16]u8" }
  ]
}
```

The capability declared in the root remains bare (`ExecObserve`); the
map referencing the imported type (`Events`) remains bare because the
map itself is declared in the root.

## Documented limits

The v0.2 scope cap is "compile-time-resolvable imports only". The
following are explicitly out of scope:

- **Remote fetching.** No `go get`-style HTTP resolution. Vendor the
  package under `./vendor/<full-path>/` instead. No lockfile, no
  checksum, no cache machinery.
- **Package versioning.** Sidestepped by vendoring. Each vendored copy
  is whatever the importing project pinned.
- **Re-exports / transitive aliasing.** Importing package A which
  re-exports package B's symbols is not supported; reference B
  directly.
- **Generic packages.** The grammar's `generic_type` covers container
  shapes only; user-defined generics over packages are not modeled.
- **Conditional compilation / build tags.** No `// +build` equivalent.
- **Private declarations.** Every top-level decl in an imported
  package is reachable from the importing package. There is no
  capitalisation rule or `pub`/`priv` keyword.
- **Per-package published manifests.** Importing a package contributes
  to the root build's single manifest; the imported package does not
  produce a standalone manifest as a side effect (though
  `hzn check ./vendor/foo` standalone still works).
- **Workspace files.** No `hzn.toml` (or equivalent) anchors a
  multi-root project; the build root is the directory passed on the
  CLI.

## Sample Migration

Suppose you have an existing single-file program that defines its
event struct and tracepoint together:

```horizon
// execcount/prog.hzn
package execcount

import bpf "m31labs.dev/horizon/runtime/kernel"

type ExecEvent struct {
    ts_ns u64
    pid   u32
    comm  [16]u8
}

map Events ringbuf[ExecEvent]

capability ExecObserve danger observe = "kernel.process.exec.observe"

@capability(ExecObserve)
@tracepoint("sched:sched_process_exec")
func OnExec(ctx tracepoint.Exec) i32 {
    event := Events.reserve()
    if event == nil { return 0 }
    event.ts_ns = bpf.ktime_get_ns()
    event.pid   = bpf.current_pid()
    bpf.current_comm(&event.comm)
    Events.submit(event)
    return 0
}
```

### Step 1: split into multiple files in one package

Move the struct into its own file. The build is unchanged — the
manifest is byte-identical.

```horizon
// execcount/types.hzn
package execcount

type ExecEvent struct {
    ts_ns u64
    pid   u32
    comm  [16]u8
}
```

```horizon
// execcount/prog.hzn
package execcount

import bpf "m31labs.dev/horizon/runtime/kernel"

map Events ringbuf[ExecEvent]

capability ExecObserve danger observe = "kernel.process.exec.observe"

@capability(ExecObserve)
@tracepoint("sched:sched_process_exec")
func OnExec(ctx tracepoint.Exec) i32 {
    event := Events.reserve()
    if event == nil { return 0 }
    event.ts_ns = bpf.ktime_get_ns()
    event.pid   = bpf.current_pid()
    bpf.current_comm(&event.comm)
    Events.submit(event)
    return 0
}
```

`examples/multifile-execcount/` is the shipped reference for this
shape.

### Step 2: extract the shared type into a vendored package

If a second program wants the same `ExecEvent` shape, move the type
into a package and vendor it under both consumers:

```
my-project/
├── prog.hzn                 # package main
└── vendor/
    └── m31labs.dev/horizon-test/events/
        └── events.hzn       # package events
```

```horizon
// vendor/m31labs.dev/horizon-test/events/events.hzn
package events

type ExecEvent struct {
    ts_ns u64
    pid   u32
    uid   u32
    comm  [16]u8
}
```

```horizon
// prog.hzn
package main

import bpf "m31labs.dev/horizon/runtime/kernel"
import events "m31labs.dev/horizon-test/events"

map Events ringbuf[events.ExecEvent]

capability ExecObserve danger observe = "kernel.process.exec.observe"

@capability(ExecObserve)
@tracepoint("sched:sched_process_exec")
func OnExec(ctx tracepoint.Exec) i32 {
    event := Events.reserve()
    if event == nil { return 0 }
    event.ts_ns = bpf.ktime_get_ns()
    event.pid   = bpf.current_pid()
    event.uid   = bpf.current_uid()
    bpf.current_comm(&event.comm)
    Events.submit(event)
    return 0
}
```

`examples/imported-execcount/` is the shipped reference for this
shape. Its golden manifest at
`testdata/golden/examples/imported-execcount/prog.cap.json` shows
`ExecEvent` emitted with `"origin": "events"` and the capability
emitted with its bare name `ExecObserve` (because the capability is
declared in the root).

### Step 3 (optional): factor capabilities into the package

A vendored package can declare its own capabilities. The aggregator
qualifies them with the import alias:

```horizon
// vendor/m31labs.dev/horizon-test/events/events.hzn
package events

import bpf "m31labs.dev/horizon/runtime/kernel"

type ExecEvent struct { ts_ns u64; pid u32; uid u32; comm [16]u8 }

capability ExecObserve danger observe = "kernel.process.exec.observe"
```

```horizon
// prog.hzn
package main

import bpf "m31labs.dev/horizon/runtime/kernel"
import events "m31labs.dev/horizon-test/events"

map Events ringbuf[events.ExecEvent]

@capability(events.ExecObserve)
@tracepoint("sched:sched_process_exec")
func OnExec(ctx tracepoint.Exec) i32 { /* ... */ return 0 }
```

The emitted manifest now carries `"name": "events.ExecObserve"` with
`"origin": "events"`. Two consumers vendoring the same `events`
package see the same qualified capability name; a third consumer
vendoring a different `events` package that ships a same-section
capability fires `HZN1553` warning, surfacing the collision without
blocking the build.
