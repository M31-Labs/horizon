# Migrating from Horizon v0.2.x to v0.3

This guide documents the user-visible changes in Horizon v0.3 and what
existing v0.2 callers should know. Each section is owned by the track
that landed the change so other tracks can append cleanly.

## HZN1564 / HZN1565 split into per-layer codes (v0.3)

Before v0.3, the codes `HZN1564` (struct / map shape conflict) and
`HZN1565` (capability / type schema conflict) fired at two different
layers — IR merge and manifest aggregation — with identical message
templates. CI logs grepping for these codes received ambiguous signal
about which layer detected the collision.

In v0.3:

- **`HZN1564`** and **`HZN1565`** now fire only at IR merge
  (`ir.MergeWithDiagnostics`), for cross-package struct-layout and
  capability collisions detected before per-package manifests are
  aggregated.
- **`HZN1566`** (map shape conflict at manifest aggregation) and
  **`HZN1567`** (type schema conflict at manifest aggregation) are new
  codes covering the post-IR-merge layer
  (`capability.AggregateManifests`).

The two layers' diagnostics also carry distinct `Suggest` strings that
name the manifest-aggregation layer explicitly, so a reader does not
need to consult code to know which layer fired.

No behavior change beyond the code split: every previously-failing
build still fails, and the diagnostic now identifies which layer
detected the conflict. CI scripts that grep for `HZN1564` / `HZN1565`
against aggregator output (for example, callers of
`capability.FromIRWithDiagnostics` or aggregator-only test fixtures)
should update their patterns to also include `HZN1566` / `HZN1567`.

The rationale for the allocation choice (why IR merge keeps the
existing codes and why the aggregator moves to the new ones) lives in
ADR-0003.

## Per-package manifest side-artifacts (v0.3)

Running `hzn check <pkg>` (where `<pkg>` is a package directory) now
emits a side-artifact `<pkg>/<pkgname>.pkg.cap.json` describing the
package's own capability declarations. The artifact is a fully-valid v1
`capability.Manifest`, produced via the same `capability.FromIR` path
that `hzn capabilities` uses. The single-rooted build-target manifest
emitted by `hzn capabilities` is unchanged.

The artifact is emitted only when the package declares at least one
capability — pure type or helper packages get no side-artifact.

Two new flags govern emission:

- **`-no-manifest`** suppresses the side-artifact entirely. Intended
  for IDE-driven `hzn check` callers that want a pure read-only check
  without filesystem mutation.
- **`-manifest-out <path>`** relocates the side-artifact to a
  caller-specified path. When set, no file is written at the default
  location.

In text mode, `hzn check` prints a new `wrote per-package manifest:
<relpath>` line after the existing `check passed: N file(s)` line so
the artifact is discoverable without filesystem walking.

CI scripts that invoke `hzn check` and inspect the working tree
afterward will see one new file per checked package. Add
`*.pkg.cap.json` to `.gitignore` if you do not want the artifact
tracked — or version-control it deliberately if your tracks include
package-scoped Continuum policy review.

The artifact filename convention is now a stable on-disk contract
consumed by external tooling (notably Continuum's per-package policy
workflow). See the Continuum integration spec §A.8 for the consumer
side. The Horizon-side decision rationale lives in ADR-0006.

## `hzn check -json` output envelope reshape [BREAKING]

The `hzn check -json` output envelope has changed shape. In v0.2 the
output was a bare JSON array of `diag.Diagnostic`:

```json
[
  { "code": "HZN3301", "severity": "error", "message": "..." }
]
```

In v0.3 the output is an object that wraps the diagnostic array and
adds a `manifest_path` field carrying the location of the per-package
manifest side-artifact (see the per-package manifest section above):

```json
{
  "diagnostics": [
    { "code": "HZN3301", "severity": "error", "message": "..." }
  ],
  "manifest_path": "vendor/foo/foo.pkg.cap.json"
}
```

`manifest_path` is `omitempty` — absent when no manifest was emitted
(suppression via `-no-manifest`, packages with zero capabilities, or
checks that exited with errors before manifest emission).

**This is a breaking change to the JSON CLI surface.** Tooling that
parses `hzn check -json` output (Continuum integrations, IDE plugins,
custom CI consumers) must update from `[]diag.Diagnostic` to the new
envelope. Minimal migration in Go:

```go
// v0.2
var diagnostics []diag.Diagnostic
json.Unmarshal(out, &diagnostics)

// v0.3
var envelope struct {
    Diagnostics  []diag.Diagnostic `json:"diagnostics"`
    ManifestPath string            `json:"manifest_path,omitempty"`
}
json.Unmarshal(out, &envelope)
diagnostics := envelope.Diagnostics
```

The text-mode output of `hzn check` is unchanged apart from the new
`wrote per-package manifest:` line documented above.

## Clang diagnostic catalog (v0.3)

Before v0.3, every clang error surfaced by `hzn diagnose` landed on
`HZN3100` (the verifier no-match sentinel) with an empty `suggest`. The
verifier-catalog origin gate skipped clang-rooted diagnostics so
verifier vocabulary did not leak into clang remediation, but the flip
side was that clang errors received no remediation either.

In v0.3, clang-rooted diagnostics flow through a new **clang catalog**
(`internal/registry/clang-catalog-v1.json`) that maps common clang
error shapes to stable `HZN34xx` codes with Horizon-flavored
remediation copy. The new range is allocated as:

- **`HZN3400`** — clang-catalog no-match sentinel. Replaces `HZN3100`
  for clang-rooted diagnostics that match no catalog entry. Like
  `HZN3100`, it carries an empty `suggest`.
- **`HZN3410`–`HZN3499`** — catalog-classified clang diagnostics, one
  code per logical class. v1 ships six seed entries:
  - **`HZN3410`** (CC0001) — undeclared `bpf_*` helper identifier
    (vmlinux.h / `runtime/horizon_bpf.h` missing).
  - **`HZN3411`** (CC0002) — generic undeclared identifier.
  - **`HZN3420`** (CC0003) — implicit function declaration.
  - **`HZN3430`** (CC0004) — unknown type name.
  - **`HZN3440`** (CC0005) — syntax error (Horizon → C lowering issue).
  - **`HZN3450`** (CC0006) — incompatible pointer / integer types
    (BPF context-pointer cast issue).

`HZN3100` semantics narrow accordingly. After v0.3 it fires **only**
for verifier-rooted diagnostics that match no verifier-catalog entry.
Pre-v0.3 `HZN3100` assertions against clang stderr migrate to either
`HZN3400` (no-match) or a specific `HZN34xx` code (catalog match).

CI scripts that pin on `HZN3100` against compile failures (clang errors
caught by `hzn build` / `hzn workbench -compile`) should be updated to
also recognise the `HZN34xx` range. The simplest pattern is to grep for
`HZN3[14]\d\d` to cover both ranges.

The catalog is content-indexed and gated by `d.Kind == "clang_diagnostic"`
in `cmd/hzn/diagnose.go`: the clang catalog is consulted only for
clang-rooted diagnostics, and the verifier catalog only for everything
else. The two catalogs are mutually exclusive by origin — no diagnostic
is ever enriched against both. Synthetic fallback diagnostics
(`Kind == ""`, e.g. raw verifier-log stdin) continue to flow through
the verifier catalog, matching v0.2 behaviour.

The clang catalog ships with a hand-crafted fixture corpus at
`testdata/clang-fixtures/<id>/{log.txt,expected.json}` mirroring
`testdata/verifier-fixtures/`. Regenerate baselines with
`make clang-fixtures-update`.

The full catalog schema and per-entry field reference live in
`spec.horizon.clang-catalog.v1`. The design rationale (regex match
style, sibling-loader pattern, `CC` prefix, range allocation including
the HZN3200 collision with bindgen that forced the move to HZN3400)
lives in ADR-0005.
