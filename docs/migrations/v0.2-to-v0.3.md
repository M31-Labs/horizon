# Migrating from Horizon v0.2.x to v0.3

This guide documents the user-visible changes in Horizon v0.3 and what
existing v0.2 callers should know. Each section is owned by the track
that landed the change so other tracks can append cleanly.

## HZN1564 / HZN1565 split into per-layer codes (v0.3)

Before v0.3, the codes `HZN1564` (struct / map shape conflict) and
`HZN1565` (capability / type schema conflict) fired at two different
layers — IR merge and manifest aggregation — with identical message
templates. CI logs grepping for these codes received ambiguous signal
about which layer detected the collision.

In v0.3:

- **`HZN1564`** and **`HZN1565`** now fire only at IR merge
  (`ir.MergeWithDiagnostics`), for cross-package struct-layout and
  capability collisions detected before per-package manifests are
  aggregated.
- **`HZN1566`** (map shape conflict at manifest aggregation) and
  **`HZN1567`** (type schema conflict at manifest aggregation) are new
  codes covering the post-IR-merge layer
  (`capability.AggregateManifests`).

The two layers' diagnostics also carry distinct `Suggest` strings that
name the manifest-aggregation layer explicitly, so a reader does not
need to consult code to know which layer fired.

No behavior change beyond the code split: every previously-failing
build still fails, and the diagnostic now identifies which layer
detected the conflict. CI scripts that grep for `HZN1564` / `HZN1565`
against aggregator output (for example, callers of
`capability.FromIRWithDiagnostics` or aggregator-only test fixtures)
should update their patterns to also include `HZN1566` / `HZN1567`.

The rationale for the allocation choice (why IR merge keeps the
existing codes and why the aggregator moves to the new ones) lives in
ADR-0003.

## Per-package manifest side-artifacts (v0.3)

Running `hzn check <pkg>` (where `<pkg>` is a package directory) now
emits a side-artifact `<pkg>/<pkgname>.pkg.cap.json` describing the
package's own capability declarations. The artifact is a fully-valid v1
`capability.Manifest`, produced via the same `capability.FromIR` path
that `hzn capabilities` uses. The single-rooted build-target manifest
emitted by `hzn capabilities` is unchanged.

The artifact is emitted only when the package declares at least one
capability — pure type or helper packages get no side-artifact.

Two new flags govern emission:

- **`-no-manifest`** suppresses the side-artifact entirely. Intended
  for IDE-driven `hzn check` callers that want a pure read-only check
  without filesystem mutation.
- **`-manifest-out <path>`** relocates the side-artifact to a
  caller-specified path. When set, no file is written at the default
  location.

In text mode, `hzn check` prints a new `wrote per-package manifest:
<relpath>` line after the existing `check passed: N file(s)` line so
the artifact is discoverable without filesystem walking.

CI scripts that invoke `hzn check` and inspect the working tree
afterward will see one new file per checked package. Add
`*.pkg.cap.json` to `.gitignore` if you do not want the artifact
tracked — or version-control it deliberately if your tracks include
package-scoped Continuum policy review.

The artifact filename convention is now a stable on-disk contract
consumed by external tooling (notably Continuum's per-package policy
workflow). See the Continuum integration spec §A.8 for the consumer
side. The Horizon-side decision rationale lives in ADR-0006.

## `hzn check -json` output envelope reshape [BREAKING]

The `hzn check -json` output envelope has changed shape. In v0.2 the
output was a bare JSON array of `diag.Diagnostic`:

```json
[
  { "code": "HZN3301", "severity": "error", "message": "..." }
]
```

In v0.3 the output is an object that wraps the diagnostic array and
adds a `manifest_path` field carrying the location of the per-package
manifest side-artifact (see the per-package manifest section above):

```json
{
  "diagnostics": [
    { "code": "HZN3301", "severity": "error", "message": "..." }
  ],
  "manifest_path": "vendor/foo/foo.pkg.cap.json"
}
```

`manifest_path` is `omitempty` — absent when no manifest was emitted
(suppression via `-no-manifest`, packages with zero capabilities, or
checks that exited with errors before manifest emission).

**This is a breaking change to the JSON CLI surface.** Tooling that
parses `hzn check -json` output (Continuum integrations, IDE plugins,
custom CI consumers) must update from `[]diag.Diagnostic` to the new
envelope. Minimal migration in Go:

```go
// v0.2
var diagnostics []diag.Diagnostic
json.Unmarshal(out, &diagnostics)

// v0.3
var envelope struct {
    Diagnostics  []diag.Diagnostic `json:"diagnostics"`
    ManifestPath string            `json:"manifest_path,omitempty"`
}
json.Unmarshal(out, &envelope)
diagnostics := envelope.Diagnostics
```

The text-mode output of `hzn check` is unchanged apart from the new
`wrote per-package manifest:` line documented above.

## Clang diagnostic catalog (v0.3)

Before v0.3, every clang error surfaced by `hzn diagnose` landed on
`HZN3100` (the verifier no-match sentinel) with an empty `suggest`. The
verifier-catalog origin gate skipped clang-rooted diagnostics so
verifier vocabulary did not leak into clang remediation, but the flip
side was that clang errors received no remediation either.

In v0.3, clang-rooted diagnostics flow through a new **clang catalog**
(`internal/registry/clang-catalog-v1.json`) that maps common clang
error shapes to stable `HZN34xx` codes with Horizon-flavored
remediation copy. The new range is allocated as:

- **`HZN3400`** — clang-catalog no-match sentinel. Replaces `HZN3100`
  for clang-rooted diagnostics that match no catalog entry. Like
  `HZN3100`, it carries an empty `suggest`.
- **`HZN3410`–`HZN3499`** — catalog-classified clang diagnostics, one
  code per logical class. v1 ships six seed entries:
  - **`HZN3410`** (CC0001) — undeclared `bpf_*` helper identifier
    (vmlinux.h / `runtime/horizon_bpf.h` missing).
  - **`HZN3411`** (CC0002) — generic undeclared identifier.
  - **`HZN3420`** (CC0003) — implicit function declaration.
  - **`HZN3430`** (CC0004) — unknown type name.
  - **`HZN3440`** (CC0005) — syntax error (Horizon → C lowering issue).
  - **`HZN3450`** (CC0006) — incompatible pointer / integer types
    (BPF context-pointer cast issue).

`HZN3100` semantics narrow accordingly. After v0.3 it fires **only**
for verifier-rooted diagnostics that match no verifier-catalog entry.
Pre-v0.3 `HZN3100` assertions against clang stderr migrate to either
`HZN3400` (no-match) or a specific `HZN34xx` code (catalog match).

CI scripts that pin on `HZN3100` against compile failures (clang errors
caught by `hzn build` / `hzn workbench -compile`) should be updated to
also recognise the `HZN34xx` range. The simplest pattern is to grep for
`HZN3[14]\d\d` to cover both ranges.

The catalog is content-indexed and gated by `d.Kind == "clang_diagnostic"`
in `cmd/hzn/diagnose.go`: the clang catalog is consulted only for
clang-rooted diagnostics, and the verifier catalog only for everything
else. The two catalogs are mutually exclusive by origin — no diagnostic
is ever enriched against both. Synthetic fallback diagnostics
(`Kind == ""`, e.g. raw verifier-log stdin) continue to flow through
the verifier catalog, matching v0.2 behaviour.

The clang catalog ships with a hand-crafted fixture corpus at
`testdata/clang-fixtures/<id>/{log.txt,expected.json}` mirroring
`testdata/verifier-fixtures/`. Regenerate baselines with
`make clang-fixtures-update`.

The full catalog schema and per-entry field reference live in
`spec.horizon.clang-catalog.v1`. The design rationale (regex match
style, sibling-loader pattern, `CC` prefix, range allocation including
the HZN3200 collision with bindgen that forced the move to HZN3400)
lives in ADR-0005.

## Private declarations: capitalization rule (v0.3 — breaking change)

Before v0.3, Horizon had no privacy model — every top-level
declaration in an imported package was accessible to importers via
the qualified-selector form `<alias>.<name>`. In v0.3, the Go-style
capitalization rule applies: a declaration is *exported* iff its
first letter is upper-case; lower-case names are *package-private*
and cannot be referenced from importing packages.

The rule applies uniformly to all top-level declarations: `type`,
`func`, `map`, `capability`, `const`.

**What breaks:** any program where one package references a
lower-case symbol in another package via the qualified-selector
form. Example: `events.execEvent` in a `map Events ringbuf[...]`
declaration, `events.helper()` in a function body, or
`@capability(events.execobserve)` on an entrypoint.

**Diagnostics:**

| Code     | Trigger                                                    |
|----------|------------------------------------------------------------|
| HZN1670  | qualified type reference resolves to a lower-case symbol   |
| HZN1671  | qualified function call targets a lower-case symbol        |
| HZN1672  | qualified map reference targets a lower-case symbol        |
| HZN1673  | `@capability(<alias>.<name>)` references a lower-case name |
| HZN1674  | qualified const reference targets a lower-case symbol      |

Each diagnostic carries a `Suggest` field that names the
capitalized form the user should rename the symbol to.

**How to migrate:**

1. **Rename the symbol to the capitalized form** in the defining
   package (preferred). The `Suggest` field on the `HZN167x`
   diagnostic spells out the recommended rename — for example,
   `execEvent` becomes `ExecEvent`. Update each call site in
   downstream packages to use the new name.
2. **Inline the call** if the cross-package usage was incidental
   and the helper has only one caller.

Same-package references to lower-case symbols remain legal —
privacy is enforced only at cross-package boundaries. A file
inside the same package that uses a lower-case helper is
unaffected.

Capitalized names continue to work as before; programs that
already use capitalized cross-package names need no change. The
`examples/imported-execcount/` example in the Horizon tree
already uses `events.ExecEvent` and is unaffected.

The rationale for choosing capitalization over an explicit
`pub`/`priv` keyword form lives in ADR-0007.

## Build tags (v0.3 — additive)

Horizon now supports file-level build constraints via
`//hzn:build <expr>` comment directives at the head of a `.hzn`
file. Files whose constraint evaluates false under the active
build context are filtered out before parsing — they contribute
no declarations and produce no diagnostics. Files with no
`//hzn:build` directive get an always-true constraint, so
existing programs are unaffected.

The expression grammar supports:

- Bare identifier dimensions: `linux`, `darwin`, `windows`,
  `freebsd`, `amd64`, `arm64`, `riscv64`, and `btf`.
- Kernel comparisons: `kernel>=5.10`, `kernel<6.0`, `kernel==5.15`,
  `kernel<=5.15`, `kernel>5.10`. Comparisons use two-part
  `<major>.<minor>` precision; patch level is ignored.
- Boolean operators: `&&` (and), `||` (or), `!` (not), with
  parenthesized grouping.

Multiple directives in the leading comment block are ANDed:

```horizon
//hzn:build linux && !arm64
//hzn:build kernel>=5.10

package monitor
```

is equivalent to a single
`//hzn:build (linux && !arm64) && (kernel>=5.10)` directive.

The active build context is detected from `runtime.GOOS`,
`runtime.GOARCH`, `uname -r`, and the presence of
`/sys/kernel/btf/vmlinux`. For cross-builds and tests, override
any dimension with the corresponding env var:

| Env var                  | Effect                                |
| ------------------------ | ------------------------------------- |
| `HORIZON_BUILD_OS`       | replaces the detected OS              |
| `HORIZON_BUILD_ARCH`     | replaces the detected architecture    |
| `HORIZON_BUILD_KERNEL`   | replaces the detected kernel version  |
| `HORIZON_BUILD_BTF`      | `1`/`true`/`yes`/`on` for true; else false |

A new diagnostic `HZN1680` fires when every file in a reached
imported package directory is excluded by the active context —
the package becomes unreachable for this build, which is almost
always a configuration error. The diagnostic suggests either
relaxing a directive or adjusting the build context.

The rationale for line-prefix syntax, the fixed dimension
whitelist, and the matcher's placement at the compiler layer
lives in ADR-0008.

## Remote imports + lockfile (v0.3 — additive)

v0.3 lets a Horizon project import packages straight from a git
repository without vendoring the source tree by hand. Vendoring still
works exactly as before — the new path is additive. Pick whichever
shape fits the project; mix freely.

### Import syntax

A remote import is any URL-shaped import path with a `@<version>`
suffix:

```hzn
import events "github.com/m31labs/horizon-test-events@v1.0.0"
```

Two version shapes are accepted:

- **Semver tag** — `@vX.Y.Z`. The leading `v` is mandatory (matches
  the `git tag` convention; bare `1.0.0` is rejected because it
  collides with branch names).
- **Git SHA** — `@<hex>` with at least 7 hex characters.

`@latest` and `@HEAD` are explicitly rejected (HZN1704) — every
build must compile to the same bits regardless of when it ran.

A URL-shaped import **without** `@version` falls through to the
existing vendor walk (`./vendor/<full-path>/`), so a team mid-
migration can flip a path at a time.

### Lockfile

Each project has one `hzn.lock` at the build root (next to the
top-level `.hzn` files). The file is JSON, sorted, and committed to
VCS — reviewers see pin changes in PR diffs:

```json
{
  "schema": "m31labs.dev/horizon/lockfile/v1",
  "entries": [
    {
      "path": "github.com/m31labs/horizon-test-events",
      "version": "v1.0.0",
      "ref_resolved": "abc1234567890abcdef1234567890abcdef12345",
      "sha256": "ffe0a0c6c33a64898da8bb9a18f6b857c9f571c3c5e7c7490c08cbe404bee625"
    }
  ]
}
```

- `version` is the user-facing pin string (semver tag or SHA prefix).
- `ref_resolved` is always the full 40-char git SHA the version
  resolved to at lockfile-write time.
- `sha256` is a content hash over the cached package's `.hzn` files
  and is verified on every `hzn check`. A mismatch fires HZN1700.
- Missing entry for a remote import fires HZN1701 with the
  suggested `hzn get <repo>@<version>` recovery command.
- Unknown schema string fires HZN1702.

### Cache

Fetched packages live under a per-user content-addressed cache:

```
$XDG_CACHE_HOME/horizon/modules/<sha256(repo)[:32]>/<ref_resolved>/
  ... package .hzn files ...
  .horizon-meta.json   (source url + fetch time)
```

If `$XDG_CACHE_HOME` is unset the cache root falls back to
`$HOME/.cache/horizon/modules/`. Two override knobs exist for tests
and advanced setups:

- `HORIZON_CACHE_ROOT` — replaces the entire cache root path.
  Useful for hermetic test fixtures (the
  `examples/remoteimport-execcount/` golden uses this).
- `HORIZON_NETWORK_TESTS=1` — opt-in flag for tests that exercise a
  real network round-trip. Default unset; `make ci-go` stays
  offline.

The cache is never committed to the project. It's per-user and
grows with the union of repos and refs you build against. Manual
`rm -rf $XDG_CACHE_HOME/horizon/modules/` is safe at any time — the
next `hzn check` against the lockfile rebuilds it.

### Workflow — `hzn get`

To add a new remote dependency:

```sh
hzn get github.com/m31labs/horizon-test-events@v1.0.0
```

This resolves the version to a SHA (`git ls-remote`), fetches into
the cache, computes the content sha256, and writes/updates the
`hzn.lock` entry. The command exits 0 on success and prints
`hzn get: added <path>@<version> -> <ref> (sha256 <prefix>...)`
on stderr. Re-running with an already-pinned version is a no-op.

Subsequent `hzn check` runs see the lockfile, find the cache entry
(or fetch on first build), verify sha256, and proceed normally.

### Co-existing with vendoring

A project can use both shapes at the same time:

- A `vendor/<path>/` directory takes precedence for URL-shaped
  imports **without** `@version`.
- A remote import **with** `@version` is resolved via the lockfile
  and cache; the vendor walk is skipped.

In practice: when migrating an existing vendored dep to the lockfile
path, the steps are (1) `hzn get <repo>@<ref>`, (2) edit the import
to add `@<ref>`, (3) `rm -rf vendor/<path>/`. The reverse migration
(remote to vendored) is symmetric.

### New diagnostics

| Code | Trigger |
| ---- | ------- |
| HZN1700 | content sha256 of cached package doesn't match lockfile pin |
| HZN1701 | remote import has no entry in `hzn.lock` — run `hzn get` |
| HZN1702 | `hzn.lock` has an unknown schema string |
| HZN1703 | fetch failed (network, auth, missing ref, etc.) |
| HZN1704 | `@<version>` is neither semver `vX.Y.Z` nor a 7+ char SHA |

### Decision rationale

The full design rationale — URL scheme choice, why JSON for the
lockfile, why the cache is sha256-of-URL keyed, the v0.3.1
fallback contract if the schedule slipped, the open-question
resolutions — lives in ADR-0009.

### Documented limits

- **URL scheme.** v0.3 wires the `github.com/<org>/<repo>` shape
  directly. A `m31labs.dev/...` meta-redirect shape (mirroring
  Go's `?go-get=1` discovery) is defined in ADR-0009 but ships as
  a stub; the production HTTP discovery surfaces in a follow-up.
- **`@latest` / `@HEAD`.** Rejected by HZN1704. Reproducibility
  is non-negotiable.
- **Auth.** Whatever git itself is configured to do (SSH keys,
  HTTPS tokens). Horizon doesn't intermediate.
- **`git+ssh://` paths.** Not in scope — too many auth shapes to
  test. Use the `github.com/...` direct path or vendor for now.
